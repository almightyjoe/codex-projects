# Parser Learning Design

The parser improves by collecting evidence, not by letting the LLM silently change code.

## Data Flow

1. `LogTailer` reads new NWN client log lines.
2. `event_parser.parse_line()` handles known formats.
3. Unknown chat-window lines are stored as `unparsed` events.
4. `DBWriter` persists them into `unparsed_lines`.
5. `learning.analyze_unparsed()` clusters repeated patterns and creates rule candidates.
6. A human reviews candidates before they are promoted into parser rules or ignored.

## Candidate Status

- `new`: generated by the analyzer, not reviewed yet.
- `approved`: accepted by a human or future review UI.
- `rejected`: not useful as a parser rule.
- `ignored`: known noise, such as console chatter or harmless cooldown messages.

## First Edge Cases Seen

- `* Mortal Strike *`
- `* Penetrating Strike *`
- alignment shifts
- potion and heal pack use
- Greater Sanctuary cooldowns
- unidentified loot notices
- armor class multi-line blocks
- console helper output
